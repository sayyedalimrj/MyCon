# Stage 10 — Evidence-Based Construction Copilot with Local Offline VLM

Stage 10 is a local Construction Copilot layer on top of the deterministic
pipeline outputs. It does **not** replace geometry, BIM alignment, deviation,
coverage, or progress computation. The VLM explains evidence generated by
Stages 7–9 and must not invent metric values.

## Local VLM policy

The default production path is local/offline:

- `copilot.vlm.provider: ollama_local`
- `copilot.vlm.endpoint: http://host.docker.internal:11434/api/chat`
- `copilot.vlm.model: qwen3-vl:8b`
- `copilot.vlm.local_only: true`

The client rejects non-local endpoints unless `allow_private_lan: true` is set
for an explicitly on-prem private server. Cloud endpoints are intentionally not
used for project data.

| SVC4 | RTX A5000 class | `qwen3-vl:8b` | `qwen3-vl:30b` | `yolo11x-seg.pt` |
| SVC2 | GTX 1080 | `qwen3-vl:4b` | `qwen3-vl:8b` | `yolo11m-seg.pt` |

Hugging Face/vLLM equivalents are available through:

- `Qwen/Qwen3-VL-8B-Instruct`
- `Qwen/Qwen3-VL-30B-A3B-Thinking`

Ollama is the simplest offline deployment. vLLM/OpenAI-compatible serving can
be used later for higher throughput.

## Evidence package

The VLM receives:

- user question,
- selected context: element, activity, bbox, view,
- rendered scan/BIM/overlay/deviation images,
- deterministic metrics JSON/CSV slices,
- limitations and confidence flags.

The VLM answer must include:

1. Direct answer.
2. Evidence used.
3. Metric facts.
4. Visual observations.
5. Confidence level.
6. Risks or uncertainty.
7. Recommended next action.

## Run Ollama fully offline after models are downloaded

On the server with internet access once:

```powershell
docker compose -f docker\docker-compose.vlm.yml up -d ollama
docker exec -it construction-ollama ollama pull qwen3-vl:8b
docker exec -it construction-ollama ollama pull qwen3-vl:30b
```

Then the model files remain in the Docker volume `ollama_models`. The server can
run without internet afterwards.

Quick check:

```powershell
Invoke-RestMethod -Uri http://127.0.0.1:11434/api/chat -Method Post -ContentType application/json -Body '{"model":"qwen3-vl:8b","stream":false,"messages":[{"role":"user","content":"Say LOCAL_OK"}]}'
```

## Run Stage 10 with local Ollama

```powershell
docker compose -f docker\docker-compose.yml run --rm core `
  python3 -m pipeline.stage_10_copilot.run_ask `
  --config configs/site01.yaml `
  --question "Has this wall been executed and can I accept it?" `
  --selected-element-id WALL-001 `
  --json
```

If Ollama is not running and `fallback_to_mock_when_unavailable: true`, Stage 10
returns a safe mock answer with a provider warning. For production acceptance,
set:

```yaml
copilot:
  vlm:
    fallback_to_mock_when_unavailable: false
    require_real_vlm: true
```

## vLLM local alternative

For a local OpenAI-compatible endpoint, set:

```yaml
copilot:
  vlm:
    provider: vllm_local
    endpoint: http://host.docker.internal:8000/v1/chat/completions
    model: Qwen/Qwen3-VL-8B-Instruct
    local_only: true
```

The code sends image evidence as data URLs and text evidence as JSON.

## What the VLM must not do

- It must not invent coverage, deviation, progress, or registration values.
- It must not replace Stage 8/9 geometry tools.
- It must not treat raw point clouds as metric truth by itself.
- If artifacts are missing, it must say which artifact is missing.

## Audit Persistence

Stage 10 now persistently logs a full deterministic audit record for every copilot interaction (when config is available).

**Saved fields:**
- Full request & answer payload
- Answer validation result
- Provider + model metadata
- Evidence package path + used evidences
- Selected element/activity/route
- Timestamp + short hash
